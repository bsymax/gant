> **文档状态**：**v0.3 业务定稿**（**§3.1 已选 A+B**，2025-04-26）。  
> **设计原则**：版本管理贯穿各阶段；由易到难迭代；在可行范围内与「目标1→8、要求1→4」的顺序对齐。

|------|------|------|

## 1. 产品概述

### 1.1 背景与问题

团队具有**探索性、创新性**，项目与事项变化频繁，许多工作需**主动发现**而非仅执行固定清单。传统重流程、轻柔性的项目工具难以同时支撑：可视化的计划与探索、人–项目关系、时间切片、能力与发展、以及面向组织绩效的决策辅助。

### 1.2 产品定位

面向 **1 名组长 + 7 名成员** 的轻量项目与能力协同系统（规模可未来扩展）：以**项目/事项**为主线，**成员**为视角，**时间**为切片，**能力**为组长度量的深度维度，在**可审计的版本历史**下协作。界面整体为**科技化、策略游戏式作战台**（指挥、资源、时间线、布阵感），**非幼龄卡通**。

### 1.3 非目标（首期明确不做或降级）

- 不替代全公司 OKR/财务系统；首版**业绩/权重**以**手工录入**为主。  
- 不承诺首版即具备复杂 AI 推荐；**规则化分析 + 可解释建议**优先，模型增强后置。  
- 首期不交付 **IM/飞书/企微/钉钉 深度集成**；仅**预留**对接与单点能力扩展点。

### 1.4 成功指标（建议，按 8 人团队校准）

| 维度 | 指标示例 |
|------|----------|
| 采纳 | 7 人成员侧周活跃 ≥ 6 人；组长周查看 ≥ 1 次 |
| 数据质量 | 关键事项更新滞后（如 >7 天未更新）占比下降 |
| 决策 | 组长反馈「能辅助排期/人力/布阵判断」的满意度 ≥ 可接受线 |

## 2. 用户与权限（对应要求2）

| 角色 | 人数 | 能力 |
|------|------|------|
| 组长 | 1 | 查看与编辑**所有人**数据；**树状能力模型、雷达、匹配分析**等**仅组长**可见可配；为成员**配置/分配**任务与目标；审批或查看变更提醒 |
| 成员 | 7（默认可扩展） | 录入与维护**与自己相关**的数据；**不暴露**能力树/能力评分界面；**修正历史已录入信息**时触发**对组长的通知/待办**；凡列入某事项的**所有关联成员**均可**编辑该事项** |

**关键规则**：

- 成员编辑：新增记录默认可选「提交后通知组长」；对**已存在记录的修正**应**显式产生一条「修正」记录**并**必达组长**（应用内 + 可扩展邮件/IM）。
- 组长调整：不强制通知全员每条变更（可配置），但建议保留**操作者与时间**的审计信息。
- **能力数据**：**树状结构**在数据层统一；**UI 上成员侧完全隐藏**（包括雷达、各叶节点分）；组长侧为「**参谋/人事面板**」类入口。产品需避免在成员端通过接口或调试轻易看到树（**后端按角色脱敏/裁剪 DTO**）。

**版本管理（对应要求1）**：

- 所有核心业务实体（项目、事项、**能力树节点引用**、个人档案关键字段、分配关系等）支持**版本/变更日志**：时间、操作者、旧值/新值或变更摘要。  
- 首版可先做「追加式历史」+ 展示；高级版本再考虑「回滚到某版本」的显式产品能力。

**对外与跨组（对应§0-6）**：

- 首版业绩：**项目级/周期级手工配置** 数值或权重。  
- **工程上**：预留 **REST/GraphQL 等扩展接口** 或**内部服务边界**（如 `TeamMetricsAdapter`、`ExternalGroupComparator` 占位实现），**不接真实多组数据**亦可通过 Mock 走通，避免后期推翻模型。

## 3. 信息架构（核心对象）

- **项目**：状态（未开展/进行中/已结束等）、时间范围、与组织绩效的关联方式（首版**手工**；**接口预留**见上）。  
- **事项/任务**（可探索发现）：状态、时间、所属项目、**所需能力**（以树**叶节点/引用**形式挂接，**仅组长看维度含义**；成员可只看到任务名与协作要求）、**所有关联人可编辑**。  
- **成员**：与事项的多对多；**能力画像**在组长端为**树形路径 + 等级**（可演进为雷达的聚合视图）。  
- **组织/团队绩效目标**（后期）：可量化 KPI；**跨组对标**走预留接口，非首版范围。

### 3.1 「未开展/意向项目」**已定稿：A + B**

**唯一产品方案（执行口径）**：

1. **储备 / 战情池（形态 B）**  
   - **新增**「未上时间轴」的意向/探索项，**默认进入储备池**（手牌/兵营/待建区），不占用时间格。  
   - 项在池内可编辑、可排序/分组；不视为已对时间有承诺。  

2. **甘特 + 未来轴 + 待定列（形态 A）**  
   - 自储备执行 **「上战场」**：用户选定**起止**或**至少开始周**后，该条**进入甘特**，出现在**未来**对应区段。  
   - 已在轴上、但**日期仍模糊**的项，显示在 **「待定 / TBD」列**（或等价的轴上虚框/泳道首列），**拖到某周/设起止**后变为**实条**。  
   - **甘特**与「本周、月至今、下一周」等**时间筛选**共用**同一时间语言**；未来负载一眼可读。  

3. **两向流转**（与 §3.1 中形态 B 流程图一致）  
   - **池 → 轴**：上战场 / 定档。  
   - **轴 → 池**：**降回探索**（从时间承诺撤回，回储备池，历史版本保留，便于解释「为什么曾上过轴」）。  

4. **实现顺序提示**：V1 可先做**池 + 上轴** + 甘特基础条带；**待定 TBD 列/泳道**若工期紧可**紧跟小版本**，但需求上属同一已定稿，不得长期只用「有日期条」而砍掉池。  

以下 **形态 A、形态 B** 为设计参考（与上文**合并为同一套体验**，非二选一）。

---

#### 参考 · 形态 A：甘特**未来时间轴**（把意向钉在或拖向时间）

**它是什么**  
- 与「本周/本月」同一套**横轴=时间**的视图上，向**右（未来）**拉一条可见的时间带；已确定起止的未开展项目/事项用**实条或半透条**画在**未来周/月**上。  
- 还没有日期的，放在时间轴**右侧「待定 / TBD」列** 或 叠在**「未来区」最左缘的虚影池**里，一拖到某周就变成「有日期的条」。

**简图（ASCII，概念示意）**

```text
时间 →     |  本周  |  下周  |  下月  |  更远  | [待定/TBD] |
-----------+--------+--------+--------+--------+------------+
项目 α     | [==== 进行中 ====]                                    |
项目 β     |        [======== 已排定未来 ========>]                |
项目 γ(意向)|                                          |  γ 虚框   |  → 拖入「下周」成实条
```

**优点**：一眼看到**主线计划与未来负载**在**同一时间语言**下对齐；和「月至今/下一周」筛选项天然连续。  
**成本**：要处理**无日期的占位**、滚动未来很远时的性能与稀疏展示（可折叠为「+N 个待定」）。

---

#### 参考 · 形态 B：储备 / 意向**列表**（与甘特**并列的「兵营/待建」面板」）

**它是什么**  
- 不占用时间格。单独「**战情 / 储备**」区域：卡片或表格列出**未开展/无承诺日期**的意向项目、探索课题、**待批准立项**。像 RTS 里**兵营里待造单位**、或**手牌区**：还没摆到时间地图上。  
- 一条意向可点「**上战场**」→ 选开始周 → 再出现在**甘特**上（或进成员布阵台）。

**简图（概念）**

```text
┌──────────── 主画面：甘特/时间线（只显示已上轴或已锁期的）───────┐
│  本周  |  下周  | ...                                              │
│  项目A ████  项目B      ████  ...                                    │
└────────────────────────────────────────────────────────────────────┘
┌──────────── 侧栏/下方：【储备 & 意向】（类似手牌/待部署单位）────┐
│  [ 意向γ ]  [ 想法δ ]  [ 下季课题ε ]  …  [拖入时间轴] [立项]     │
└────────────────────────────────────────────────────────────────────┘
```

**优点**：**探索/模糊**的不会污染时间轴；脑暴与承诺分层，符合「要去找事情」的用法。  
**成本**：需一条明确交互「从储备 → 上轴」，否则两处以谁为准容易分歧。

**流程（形态 B 常配合）**：

```mermaid
flowchart LR
  subgraph pool [储备池]
    I[意向/探索项]
  end
  subgraph timeline [时间轴甘特]
    T[有起止的条带]
  end
  I -->|上战场 / 定档| T
  T -->|降回探索| I
```

#### 与已定稿的对应关系

| 能力 | 来自 |
|------|------|
| 池子养探索、不污染时间 | **B** |
| 上轴后未来区 + 与筛选器一致 | **A** |
| 待定 TBD 处理未钉死日期的在轴项 | **A** |
| 池 ↔ 轴可逆、承诺分层 | **A+B 组合**（上文「唯一产品方案」） |

## 3.2 与策略游戏/沙盘布阵的对应（对应§0-7 风格）

- **主视图**：甘特/时间 + **「排兵布阵」**：项目/事项为**单位格**，成员为**可指派实体**，拖拽到项目槽位即人力分配（实现顺序可在 V1 简化为**列表指派**，V2+ 增强拖拽与冲突提示）。  
- **语汇可包装**：指挥部、作战序列、待部署、战线、补员（不替代清晰业务文案，**设置里可开「军语模式」** 娱乐向可选）。  
- **动效与材质**：微光、网格、扫描线、细边框、危险/超载红边（人力过载时），**克制**不干扰读写。

## 4. 分版本规划（由易到难，与目标顺序对齐）

下列版本为**建议切片**；可合并或拆分发布，以迭代节奏为准。

### V1 — 基座：项目进展与类甘特视图（目标1）+ 版本化数据与角色分工（要求1、2）+ 策略科技风 UI（要求3 初版）

**范围**：

- 项目列表与状态；**§3.1 已定稿 A+B**：**储备/意向池** + **含未来与待定 TBD 列的甘特** + **上战场 / 降回探索** 流转。  
- 项目–事项（任务）两级结构；事项有基础字段（名称、起止、状态、**全部关联人可改** 的**协作/负责人**关系）。  
- 用户模型：**1 组长 + 7 成员**（数据模型允许后续扩展人数）。**本地部署**优先。  
- 登录后角色分流：**组长**与**成员**的菜单与写权限见§2；**成员端不出现能力树/雷达**（V1 可不实现能力，仅做权限与壳）。  
- **变更历史/版本记录**在事项与项目维度可用（最低限度可读）。  
- **视觉**：**策略/RTS 指挥台**气质（见§3.2），网格、线框、轻 HUD；为后续**沙盘布阵式排人**留交互位（V1 可先用表单指派）。

**验收**：组长能使用**储备池**与**甘特未来+TBD** 的完整组合；**上战场/降回探索** 可用；成员能维护自己相关事项、协作人**协同编辑**；修改历史可查证；可本地安装运行（若首版分「池+上轴+基础甘特」与「TBD 列」两小步交付，**文档验收以 §3.1 全文为准、拆分子里程碑在 changelog 中说明**）。

**依赖/风险**：无。

**不纳入本版**：与 §0-8 一致的**外部 IM/单点**集成。

---

### V2 — 成员维度：在做什么/已做/将做（目标2）

**范围**：

- **成员看板/泳道**（或按人聚合的列表 + 三栏）：进行中 / 已完成 / 待办（或排期中）。  
- 显式 **独自承接 vs 合作承接**（多负责人 + 协作者、或主责 + 协作者模式）。  
- 组长为成员**配置任务/目标**的入口（可落到具体事项或目标包）。

**验收**：选人即可过滤其全量或时间范围内 workload。

**依赖**：V1 的项目–事项与成员关系。

---

### V3 — 时间切片：本周 / 月至今 / 下一周 等（目标3）

**范围**：

- 全局**时间范围快捷筛选**：本周、本月至今日、下一周、自定义区间。  
- 上述筛选作用于：甘特、成员维视图、项目列表与统计小计。  
- 与 **§3.1 A+B** 配合：选未来区间时，高亮**落在该窗内的甘特项**；可选筛选「本窗内**自储备池上战场**转入的项」以便复盘承诺节奏。

**依赖**：V1、V2 的日期与状态数据完整；**未开展/意向** 的数据来源与 **§3.1** 一致。

---

### V4 — 能力（树状，仅组长）+ 雷达/聚合视图 + 游戏化（目标4、要求3 强化）

**范围**：

- **树状能力库**（可配置层级：领域 → 子项 → 叶技能）；**事项挂接叶/节点**；**全部能力维护与能力–人评分仅在组长端**。  
- 组长端：成员**能力雷达或树折叠面板**、多成员**对比**；与事项**需求–供给**简版**缺口/超载**提示。  
- **游戏化**（与 §0-7 一致，偏**策略游戏**感）：**XP/等级/勋章**等可二选一或组合，规则在组长端**显式**；成员端**仅**展示**不泄露树结构**的反馈（如「本任务贡献」「本周战功」等文案级包装，**具体字段产品再细定**，避免和「隐藏能力树」冲突）。  
- 成员**不自评**能力树节点；若需反馈走**任务/成果回写** 或 组长**录入/校准**（定稿以「仅组长见树」为准）。

**依赖**：V2、V3 稳定的事项数据。

**接口与扩展**：能力–事项–人数据通过 **API/DTO 按角色脱敏**；**业绩与跨组**仍走 §2 的预留边界。

---

### V5 — 长期成长与培养计划（目标5）

**范围**：

- 能力随**时间**变化曲线（需定期快照或重要事件落库）。  
- **培训/培养计划**：计划项、目标能力、起止、与项目/任务的可选关联。  
- 与组长协同：计划审批或「知会」流（可简化为评论 + 状态）。

**依赖**：V4 能力模型。

---

### V6 — 项目分析与团队产出视角（目标6 主体）

**范围**（建议全部**可解释、可关闭**）：

- 项目级：**人天/人力占用粗算**、进度风险标记、**性价比启发式**（如进度/投入比、**手工配置**的价值/权重参与计算）。  
- 建议类输出（非命令）：**减少人手 / 增派人手 / 人岗不匹配** 等，附**依据**；**人岗/能力侧依据仅组长**可见。  
- **业绩**：**首版及延续多版以手工配置为主**；**接口预留** 供后续接入其他组/组织数据、横向比较（**Mock 可先行**）。

**依赖**：V3 时间、V2 人力结构；V4 为「不匹配」类建议提供**组长侧**能力数据。

**已定稿要点**：不强制在首版锁定「业绩单位」口径；以**可编辑字段 + 可解释公式**逐步收紧。

---

### V7 — 人岗错配、培养与交付的平衡（目标7）

**范围**：

- **拉伸任务（Stretch）** 与 **项目质量/截止风险** 双轴评分。  
- 策略选择（组长配置）：如「本季度偏培养」vs「本季度保交付」影响建议权重。  
- 在事项或项目层展示**匹配度+风险提示**，避免单一数值绑架决策。

**依赖**：V4、V5、V6 数据与规则基础。
